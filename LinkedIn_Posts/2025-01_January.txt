But what makes them special? Here's the scoop on Static Pods: What are Static Pods? Static Pods are directly managed by the Kubelet, not the Kubernetes API server. They are tied to a specific node and don't rely on higher-level controllers like Deployments. When are Static Pods used? They are commonly used for essential system-level processes, debugging, or components like the Kubernetes control plane itself. How are Static Pods created? Instead of defining them through kubectl or a manifest applied to the API server, Static Pods are defined in the staticPodPath directory configured in the Kubelet. Example location: /etc/kubernetes/manifests/ Any .yaml file dropped into this folder is automatically converted into a pod. Key Benefits: Simple to set up for node-specific applications. Independent of the control plane, so they can run even if the API server isn't available. Essential for Kubernetes bootstrapping (e.g., control plane components like kube-apiserver). Limitations: No direct management via the Kubernetes API. Manual scaling and updates are required. Limited to single-node use. #DAY132 #Labels vs. #Service Selectors in Kubernetes In Kubernetes, everything revolves around YAML manifests. A resource's metadata describes the resource itself (like a Pod, Deployment, or Service), while the spec defines how it should behave. Labels in metadata These are key-value tags you attach to an object. For example: metadata: labels: app: my-app environment: production They help organize, group, and search your resources. You can run kubectl get pods -l app=my-app to find all Pods labeled app=my-app, for instance. #Service Selectors A Service uses spec.selector to find the Pods it routes traffic to. It looks for Pods whose labels match the selector: spec: selector: app: my-app This means the Service's own labels (metadata.labels) are purely for identifying the Service itself. They do not decide which Pods receive traffic. Why This Matters Decoupling how you label the Service from the labels on Pods lets you organize resources differently and maintain flexibility. #Summary #metadata.labels = "I am a resource with these labels." #Service's spec.selector = "I want to find Pods that have these labels." Remember: Services match Pods by spec.selector, not by the Service's own labels.
