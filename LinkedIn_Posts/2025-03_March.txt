It can enforce security policies, restrict permissions, and analyze service risks. A key tool for this is systemd-analyze security, which evaluates services and assigns an "exposure" score based on security settings. Example output: systemd-analyze security nginx.service SERVICE EXPOSURE DETAILS nginx.service 6.8 No ProtectSystem, No NoNewPrivileges The higher the exposure score, the more security risks exist. How This Connects to Kubernetes Kubernetes runs workloads inside containers, but those containers still rely on Linux. Systemd's security settings directly impact how containers inherit privileges and interact with the host system. A common mistake is running containers with excessive privileges, allowing them to bypass system security policies. #DAY166 Kubernetes Readiness & Liveness Probes: A Simple Fix for Big Problems Ever had a pod in Kubernetes that looks like it's running but isn't actually working? That's where Readiness & Liveness Probes come in—two simple yet powerful features that can prevent unnecessary downtime. Liveness Probe checks if the container is alive. If it fails, Kubernetes restarts the pod. Readiness Probe checks if the container is ready to receive traffic. If it fails, traffic is temporarily stopped until it recovers. A common issue is microservices that take a while to start. Without a Readiness Probe, Kubernetes might start sending traffic too soon, leading to failed requests. A simple probe ensures traffic is only routed when the service is actually ready. Here's a quick example: livenessProbe: httpGet: path: /health port: 8080 initialDelaySeconds: 5 periodSeconds: 10 readinessProbe: httpGet: path: /ready port: 8080 initialDelaySeconds: 10 periodSeconds: 5 It's a small configuration change that makes a big difference in keeping applications stable and responsive.
