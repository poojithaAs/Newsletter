ID:7323345432779116544 #DAY211 Containerd CLIs: From "Why is this so hard?" to "I am the orchestrator now." When you move from Docker to containerd, no one warns you… You lose docker ps, docker run, docker logs, and even your will to live. Instead, you're handed three mysterious tools: • crictl • ctr • nerdctl And you're left Googling like your career depends on it. Let's decode the madness. 1. crictl: The Pod Investigator "I'm just here to check what the kubelet is doing, ma'am." What it is: A CLI tool to talk to container runtimes (via the Kubernetes CRI). Great for: • Listing pods, containers, images on your node • Debugging CRI-level issues (kubelet ↔ containerd) • Running low-level commands like crictl pods, crictl inspect Not for: • Building images, running containers like Docker crictl ps crictl inspect <container-id Tip: If your pod is failing and you can't kubectl exec, crictl is your CSI. ctr: The Sysadmin's Secret Weapon "You want to run containers? Sure,if you know the magic incantation." What it is: The native CLI for containerd. Extremely powerful. Zero patience. Great for: • Debugging containerd internals • Low-level container lifecycle management Pain points: • No human-friendly help messages • Non-intuitive syntax • No volume, build, or compose support ctr images pull https://lnkd.in/gQKYjX6R ctr run --rm -t https://lnkd.in/gQKYjX6R Pro tip: If you're using ctr, just know: you're in sysadmin mode now. 3. nerdctl: The Dev's Best Friend "Hey Docker folks. I gotchu." What it is: A Docker-compatible CLI for containerd. Think: docker but for containerd. Great for: • Docker-style run, build, logs, exec • compose support via nerdctl compose • Working with K3s (which uses containerd under the hood) Why people love it: • Familiar syntax • Works with rootless mode • Supports CNI plugins, BuildKit, etc. Tip: If you want Docker features without Docker running • crictl for Kubernetes • ctr for hardcore internals • nerdctl for real-world dev sanity Because YAML may be our religion, but we still deserve nice things.

ID:7322675653466619905 #DAY209 Is Jenkins Still Relevant Today? Let's Talk About It. When Jenkins first emerged in 2011 (building on the Hudson project), it revolutionized CI/CD. • Before Jenkins, automating builds and deployments was a manual, painful task. • Jenkins popularized the concept of Continuous Integration, making it accessible to teams everywhere. • It was open-source, customizable through plugins, and flexible enough to fit almost any environment — from startups to Fortune 500 companies. In many ways, Jenkins defined DevOps pipelines before "DevOps" even became a buzzword. Fast forward to today. The world has changed: • Infrastructure has moved to the cloud. • Workloads are increasingly containerized and Kubernetes-native. • We expect elastic scaling, immutable deployments, and self-healing systems — things that weren't the design priorities when Jenkins was born. While Jenkins is still powerful, it struggles with: • Scaling pain: Managing Jenkins clusters at cloud-scale can be complex. • Plugin sprawl: Too many plugins create fragility and maintenance nightmares. • Modern developer experience: Tools like GitHub Actions, GitLab CI/CD, ArgoCD, and Tekton provide faster, lighter, and more integrated workflows — especially in cloud-native environments. So, is Jenkins dead? Not at all. • It remains relevant in many organizations with large, customized legacy pipelines. • It's still hugely flexible and battle-tested. • However, for new, cloud-native projects, many teams are choosing tools that are built for scale, speed, and Kubernetes from the ground up.

#DAY204 #TerraformTales: The Mystery of the Missing S3 Bucket Problem: You apply your Terraform config, it runs successfully. But the S3 bucket? Nowhere to be found. You check the state. You double-check the region. You triple-check the bucket name. Still… nothing. Turns out—you used a count to conditionally create the bucket: resource "aws_s3_bucket" "logs" { count = var.enable_logging ? 1 : 0 bucket = "my-app-logs" } But in your variables? enable_logging = false. So Terraform silently skips creating the bucket. No errors, no warnings, just… nothing. Solution: Add output visibility or a warning to avoid getting ghosted by your own config: output "logging_enabled" { value = var.enable_logging description = "Is logging enabled for this environment?" } Or, better yet—use terraform plan and actually read it (yes, we're all guilty). Lesson: Just because your plan is green doesn't mean your infra is real. Terraform will always do exactly what you tell it—nothing more, nothing less.
