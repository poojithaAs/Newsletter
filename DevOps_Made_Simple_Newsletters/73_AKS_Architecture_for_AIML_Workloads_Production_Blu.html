AKS Architecture for AI/ML Workloads (Production Blueprint)

✍️ By Poojitha A S | DevOps | SRE | Kubernetes | Reliability Engineering

"AI workloads don't fail like normal apps. They fail expensively."

If you run AI/ML on AKS, the fastest way to burn money is simple: You mix everything on the same nodes and hope autoscaling will save you.

It won't.

What will save you is a clean architecture: right-sized node pools + strict scheduling + intelligent autoscaling + GPU discipline.

Kubernetes = Your AI City Infrastructure

Think of your AKS cluster like a smart city.

  • Buildings (Pods): where your models and services live

  • Neighborhoods (Namespaces): dev, staging, prod separation

  • Roads (Services + Ingress): how users reach your AI

  • Power Plants (GPU Nodes): specialized compute for inference/training

  • Warehouses (Persistent Volumes): datasets, model artifacts, checkpoints

  • Traffic Control (Network Policies): isolation and security boundaries

  • City Planning (HPA/VPA/KEDA): scaling with demand

When the city is designed well, traffic flows. When it's not, everything jams… and you pay for the chaos.

Kubernetes Architecture for AI/ML: The 3-Tier Node Pool Strategy

  1. System Pool (Cluster-Critical Only)The foundation. Keeps the cluster stable.VM: Standard_D4s_v3 (4 vCPU, 16GB) Nodes: 3 (for high availability) Runs: monitoring, system components, control-plane add-ons Rule: No app workloads here, ever.

  2. CPU Pool (General Workloads)Where most of your AI platform runs.VM: Standard_D8s_v3 (8 vCPU, 32GB)Autoscaling: 2 → 10 nodes Runs: APIs, preprocessing, embeddings, feature services, batch workersGoal: Keep this pool boring. Boring is good in production.

  3. GPU Pool (Training + Inference)The power plant. Scale to zero when idle.VM: Standard_NC6s_v3 (V100) Autoscaling: 0 → 5 nodes Runs: inference, fine-tuning, heavy embedding jobs, GPU batch pipelinesRule: Only GPU workloads land here.Tip: Use taints/tolerations to enforce this. This one choice is the difference between controlled spend and a $12K bill.

Autoscaling Strategies That Actually Work: Autoscaling has three layers.

  1. Cluster Autoscaler (Node Scaling): Adds nodes when pods can't fit. Removes empty nodes. This is your foundation.

  2. HPA (Pod Scaling): Scales pods based on CPU, memory, and real signals like requests per second, queue depth, latency, and concurrency. CPU alone is not enough for AI—you'll scale too late and too expensively.

  3. KEDA (Event-Driven Scaling): Scales based on events and queues. Lets you run zero replicas when idle, so you pay nothing when not in use. Perfect for bursty AI workloads.

GPU Resource Management (Where Most Teams Lose Money)

GPU nodes are not expensive because they exist. They are expensive because they sit idle.

Requests and Limits (Non-Negotiable)

If you don't set GPU requests properly, you don't control scheduling.

GPU Sharing (Only When It Makes Sense)
Two common approaches:
  • Time-slicing / NVIDIA MPS: share a GPU across multiple pods
  • MIG: split a single A100/H100 into multiple GPU instances Note: MIG is for A100/H100 class GPUs, not V100.
GPU sharing is powerful. But only if you track latency and throughput carefully.
Cost Optimization Patterns 
1) Spot Instances for Fault-Tolerant Jobs
Batch inference, async pipelines, retryable tasks.
If it can be interrupted, it should not be on-demand.
Savings are real. Interruptions are also real. Design for both.
2) Just-in-Time Provisioning (Right Nodes at the Right Time)
The best clusters don't "keep extra nodes warm." They provision exactly what's needed when a workload appears.
This is how you avoid paying for idle GPU.
Monitoring & Observability (AI Needs More Than "CPU is fine")
If you monitor AI like a normal microservice, you'll miss the real failures.
Track:
  • GPU utilization and memory usage
  • inference latency (p50, p95, p99)
  • scheduling success rate (pending pods)
  • node provisioning time
  • disk IO and network throughput
  • model quality signals (accuracy, drift, degradation)
AI reliability is not only uptime. It's also "is the model still correct?"
TL;DR
  • Separate workloads using 3 node pools: system, CPU, GPU
  • Use autoscaling in layers: cluster autoscaler + HPA + KEDA
  • Treat GPUs like money machines: schedule strictly, scale-to-zero, monitor deeply
  • Cost control comes from architecture, not from "trying to optimize later"
Key takeaway: Right-sized node pools + intelligent autoscaling can unlock 40–60% infrastructure cost savings while still meeting performance SLAs.
