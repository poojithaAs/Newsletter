The Quiet Money Leak

Your Kubernetes cluster is burning money right now. Not in a dramatic way. Not in a way that triggers alarms. In a quiet, 20–40% overspend way. 68% of teams have this problem.

This is not a scaling issue. This is not a capacity planning failure. This is negligence.

And I'm about to show you how to fix it in 10 minutes.

Why K8s Cost Waste Is Getting Worse

Three years ago, Kubernetes cost waste looked like oversized node pools and idle stateful sets. You could spot the problems visually.

Then AI happened.

GPUs changed the equation. A100s cost $2–3/hour. V100s cost $1.50/hour. When you're running them at 10% utilization waiting for batch inference requests, you're hemorrhaging money.

Add to that:

  • Development clusters running 24/7: No one turns them off on weekends. They cost 30% more than they should.
  • Reserved instances expiring: Teams buy 1-year RIs, forget about them, lose 20% discount when they renew.
  • Cross-team resource requests: Nobody knows who owns what. Zombie deployments run for months.
  • Monitoring and logging overhead: Shipping logs to every vendor doubles your egress costs.

Real story: One SRE team found 3 A100 GPUs running continuously for 18 months under a "test" namespace. Cost: $47,000. They never checked.

The 10-Minute Audit

Five steps. Two minutes each. You'll find money leaks worth thousands.

1
Check Resource Requests vs Actual Usage

Your pod requests 2 CPUs and 4GB RAM. It uses 0.5 CPU and 200MB RAM. Multiply that across 500 pods and you're paying for 4x more than you need.

2
Find Zombie Deployments

Deployments with 0 replicas still exist. Helm releases you forgot about. Stateful sets reserving resources.

3
Audit Node Pool Utilization

GPU nodes running at 10% utilization. CPU-only nodes in a GPU cluster. Reserved capacity for peak traffic that never comes.

4
Check for Missing Autoscaling

HPA (Horizontal Pod Autoscaling) disabled. Karpenter not installed. KEDA jobs running on static resources.

5
Review Spot/Preemptible Usage

Production running on on-demand. Non-critical jobs paying for guaranteed uptime. Savings: 60–90%.

Step-by-Step Walkthrough

Step 1: Resource Requests vs Actual Usage (2 minutes)

kubectl top pods --all-namespaces | head -20

This shows actual CPU/memory usage. Now check what was requested:

kubectl get pods --all-namespaces -o jsonpath='{range .items[*]}{.metadata.namespace}{"\t"}{.metadata.name}{"\t"}{.spec.containers[0].resources.requests}{"\n"}{end}' | head -20

Compare the two. If requests are 5x actual usage, you found waste. Count how many pods have this issue. Multiply by your hourly node cost. That's money you're leaving on the table.

Step 2: Zombie Deployments (2 minutes)

kubectl get deployments --all-namespaces | grep "0/0"

Any deployment showing 0 desired replicas, 0 running replicas? Delete it. If you can't delete it (compliance), scale it to zero explicitly and set a reminder to remove it in 30 days.

kubectl get ns --sort-by=.metadata.creationTimestamp | tail -10

Check which namespaces were created recently. Are they still needed? Helm releases from 2 years ago still running? Time to clean house.

Step 3: Node Pool Utilization (2 minutes)

kubectl describe nodes | grep -A 3 "Allocated resources"

This shows what % of each node's CPU/memory is actually in use. If you see nodes at 15% utilization, you're paying for empty capacity. If you have GPU nodes, check nvidia-smi:

kubectl exec -it -- nvidia-smi

If GPU utilization is under 20%, you're wasting $20/day per GPU. NVIDIA MIG (Multi-Instance GPU) can save 85% on these workloads. Not configured? You're bleeding money.

Step 4: Missing Autoscaling (2 minutes)

kubectl get hpa --all-namespaces kubectl get keda-scaledobjects --all-namespaces

If these return empty, you don't have autoscaling. Batch jobs, background workers, and CI/CD nodes should scale to zero when idle. If they don't, add HPA or KEDA immediately.

Step 5: Spot vs On-Demand (2 minutes)

kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.metadata.labels.karpenter\.sh/capacity-type}{"\n"}{end}'

Spot instances cost 60–90% less. Non-critical workloads (batch, logging, CI/CD) should run on spot by default. Production critical? Use on-demand. But be intentional. Right now you're probably defaulting to on-demand everywhere.

Making Cost Discipline Stick

One audit is good. Systemic change is better. Treat cost policies like you treat security policies:

  • Resource request enforcement: Policy: all pods must define CPU and memory requests. Violations blocked by admission controller.
  • Autoscaling requirements: Batch jobs must have HPA or KEDA. No exceptions.
  • Spot-first strategy: Non-critical workloads default to spot. Opt-in to on-demand, don't opt-out.
  • Monthly cost reviews: Tie it to infrastructure metrics. Track trends. Alert on anomalies.
  • Cloud spend tagging: Allocate costs to teams. They'll police themselves.

Pro tip: Set up cost dashboards in your monitoring tool. Make team cost visibility automatic. When engineers see they're burning $5K/day on their cluster, requests drop magically.

Day 1–5 Action Plan

  • Day 1: Run the audit. Document findings in a spreadsheet. Identify top 5 waste sources.
  • Day 2–3: Kill zombie deployments. Resize resource requests for top 10 deployments.
  • Day 4: Propose GPU consolidation plan (NVIDIA MIG or dedicated inference clusters).
  • Day 5: Set up cost dashboards. Brief leadership on findings and savings potential.

Expected savings: $2–5K/month in the first week. Compounding savings of 15–25% within 90 days.
