This edition breaks down the 3 pillars of modern traffic management:

  • Load Balancing

  • Autoscaling

  • Load Shedding

All working together, not in silos.

What's Load Management?

Managing load means adjusting your system's capacity and routing to match incoming demand.

But traffic isn't polite. Clients retry. Regions fail. Code bugs cause loops.

So modern systems need:

  • feedback from infra

  • Coordination between scaling, balancing, and shedding tools

  • Room to absorb spikes (with fallback plans)

🧠 Case Study: Pokémon GO Crushes GCLB

What Happened?

Niantic planned for 5x traffic at launch.

They got 50x.

Their original setup—using regional NLBs and Nginx proxies—choked during a global SYN flood and synchronized client retries.

The Fix?

Google helped Niantic migrate to GCLB (Google Cloud Load Balancer):

  • Anycast IP + Global Maglev Load Balancer

  • GFEs (Google Front Ends) close to users

  • Auto-scaling backend VMs and draining failed ones

  • Load-shedding throttles to prevent full collapse

🧨 Key Failure: Load shedding caused clients to retry in sync

💡 Lesson: Retry storms + unhealthy backends = cascading failure

Load Balancing: Not Just DNS Anymore

Google's GCLB uses Anycast + Maglev instead of DNS.

Why?

  • DNS caching is unpredictable

  • Anycast routes to nearest healthy cluster

  • Maglev uses consistent hashing + connection tracking

  • GFEs health-check and reroute around broken backends

💡 Pro Tip: Stable Anycast = fewer resets when routes flap

🛡️ GFEs enter "lame duck mode" to gracefully retire during updates

Autoscaling: Smart... Until It's Not

When done right, autoscaling reacts before your service breaks.

✅ Best Practices:

  • Use load balancer metrics, not CPU alone

  • Wait for cool-down periods before counting new instances

  • Combine with autohealing to restart zombies

⚠️ Misuse Scenarios:

  • Scaling up endlessly due to CPU bugs

  • Scaling up stalled requests stuck on failing dependencies

  • Crashing other microservices due to shared quota

📉 Solution: Always set limits (min/max instances), define quotas, and include manual kill switches

Stateful Systems? Horizontal Scaling ≠ Help

If your system holds sessions:

  • Load can't just jump to new instances

  • Add task-level routing (e.g., consistent hashing)

  • Use vertical scaling with care—it can over-provision idle paths

🧵 Case Study: Dressy's Broken Balance

What Happened?

A dress shopping app "Dressy" ran across 3 regions. As traffic grew, Region A started load shedding due to CPU. But their autoscaler saw less CPU per request and assumed A was more efficient.

Result? More traffic sent to the failing region. Users got errors. Other zones sat idle.

Root Cause?

Load shedding logic didn't inform the load balancer.

💡 Fix: Count failed requests as 120% CPU to signal distress

🔧 Lesson: All systems (autoscale, shed, balance) must communicate

Combining Strategies Like a Pro

When do you need all three?

✅ When traffic is bursty

✅ When serving from multiple regions

✅ When failover must be instant

✅ When backend services have nonlinear scaling

Examples:

  • Run microservices in zonal clusters with cross-regional autoscaling

  • Add load shedding only after autoscaling triggers

  • Use deadlines for RPCs to prevent memory overloads

Managing Load in gRPC Systems

Set request deadlines.

Otherwise:

  • Resources are held too long

  • Clients experience high latency

  • Server memory crashes under load

🛠️ Tip: Use .proto file comments to define defaults

🧏 Let servers cancel long-running requests to free resources

TL;DR: Managing Load = Managing Relationships

✅ Load Balancing = Spread smartly

✅ Autoscaling = Prepare for spikes, recover from dips

✅ Load Shedding = Buy time, not blame

But here's the kicker:

If they don't work together, they'll fight each other.

And the result? Dressy. Pokémon GO. Your next incident.
