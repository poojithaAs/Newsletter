"Great systems aren't just drawn :they're designed to work in the real world."

Have you ever created a design that looked perfect on a whiteboard, but didn't survive real-world traffic?

Google has. That's why they use a method called:

👉 Non-Abstract Large System Design (NALSD)

Let's explore how it works and why it's one of the most important skills for anyone building reliable systems.

💡 What is NALSD?

NALSD is about designing systems that work in real life, not just in theory.

It asks us to think about:

  • What happens when the system grows?

  • What if something fails?

  • How much memory, storage, or machines will we need?

Instead of assuming things will work, NALSD encourages testing and adjusting the design at every step.

🧠 The 4 Key Questions of NALSD

Every time we design or improve a system, we ask:

  1. Is it possible? Can we actually build this system?

  2. Can we do better? Is there a simpler or faster version?

  3. Is it practical? Does it fit our time, budget, and tools?

  4. Is it reliable? Will it still work if something goes wrong?

We repeat these questions as we refine the design.

Case Study: Google AdWords CTR

Google needed to calculate the Click-Through Rate (CTR) of ads in real time.

Requirements:

  • Handle 500,000 searches and 10,000 clicks per second

  • Show results in less than 1 second

  • Keep data less than 5 minutes old

Now let's see how they built a system that meets these goals.

🧡 Step 1: One Machine for Everything?

They first tried putting everything on a single machine.

❌ Not enough memory or storage

❌ Couldn't handle the traffic

❌ If that machine failed, the whole system went down

➡️ Lesson: One machine is a risk. We need to scale out.

🗃️ Step 2: Try MapReduce (batch processing)

MapReduce was used to join the logs and calculate CTRs.

✅ Scales well

❌ Not fast enough for real-time results

❌ Click and search logs might land in different batches

➡️ Lesson: Works for reports, not live dashboards.

🔗 Step 3: Build LogJoiner and QueryStore

They created a better design:

  • QueryStore holds all search data

  • LogJoiner joins clicks with the matching searches

  • ClickMap & QueryMap organize results by ad_id for quick lookup

✅ Real-time processing

✅ Fast lookups for dashboard

➡️ Lesson: Separate storage and processing makes things faster and cleaner.

🤩 Step 4: Use Sharding for Scale

They split the logs into smaller parts, or shards, and ran many LogJoiner instances.

  • Each instance handles only part of the data

  • System continues working even if one machine fails

  • ClickMap and QueryMap are also sharded and duplicated

➡️ Lesson: Scaling horizontally helps the system grow and stay reliable.

🌐 Step 5: Add Multiple Data Centers

What if an entire data center fails?

Google:

  • Duplicated data in 3+ locations

  • Used a system called Paxos to keep everything in sync

  • Ran thousands of small tasks across machines to handle load

✅ Even if one region fails, others can take over

✅ Consistent and safe data writes

➡️ Lesson: True reliability means designing for major failures, not just small ones.

🧥 TL;DR

  • NALSD helps us build systems that can survive real-world usage

  • Start simple, then ask:

  • – Can we build it?

  • – Can we improve it?

  • – Can we afford it?

  • – Can it handle failures?

  • Each design round teaches us more

  • Scale with sharding, and build for failure from the start
