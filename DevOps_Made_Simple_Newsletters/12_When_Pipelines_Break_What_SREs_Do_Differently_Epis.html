Small failures snowball fast in complex pipelines. Prevention starts with visibility.

Pipelines are everywhere they power dashboards, ML models, user metrics, payments, and more. But if one failure takes them down, the cost can be huge.

This chapter is about how Google and Spotify approach pipelines with SRE best practices. It breaks down:

  • Checkpointing

  • Reusability

  • Failure handling

  • SLOs

  • Maturity models

  • Spotify's real-world architecture

🔄 Checkpointing: Resume, Don't Restart

Long-running pipelines often crash or are rescheduled.

Without checkpoints, one failure means starting from scratch.

Checkpointing allows pipelines to save state regularly:

  • If the pipeline stops, it can pick up from the last checkpoint.

  • Saves time, cost, and unnecessary reprocessing.

  • Works well for pipelines training ML models, processing large logs, or operating across sessions.

It also helps during rescheduling (e.g., when changing CPU/memory).

📦 Reusable Pipelines = Easier Scaling

"Write once. Reuse everywhere."

When you have 10+ pipelines doing similar tasks, adding a new metric or monitoring logic across all of them can be painful.

Instead:

  • Create shared libraries for metrics, alerting, logging, or input parsing.

  • Use common building blocks.

  • Alert once across many jobs.

Think microservices-style pipelines: small, focused components doing one job well.

📊 Pipeline Maturity Matrix: Google's Checklist

Google created a maturity matrix to evaluate pipeline reliability across five categories:

1. Failure Tolerance

  • No retries? Score: 1

  • Auto-retries? Score: 3

  • Multi-region failover with automatic recovery? Score: 5

2. Scalability

  • Manual scale? Score: 1

  • Auto-scaling with configs? Score: 3

  • Load shedding, priority queues, dynamic sharding? Score: 5

3. Monitoring & Debugging

  • No logs? Score: 1

  • Basic error tracking? Score: 3

  • Full dashboards, auto-quarantine, replay systems? Score: 5

4. Testing

  • No tests? Score: 1

  • Unit tests only? Score: 3

  • Unit + integration + end-to-end with auto-validation? Score: 5

5. Ease of Use

  • No discoverability or docs? Score: 1

  • Basic API docs? Score: 3

  • Auto-discoverable, machine-readable configs, training kits? Score: 5

Use this to identify weak areas and track improvements.

⚠️ Common Pipeline Failure Modes

  1. Delayed Data Downstream jobs start too early – before all input is available. Result: stale or wrong output. Fix: enforce data dependencies, pause if inputs are missing.

  2. Corrupt Data Software bugs, config errors, or bad input cause invalid outputs. Fix: block bad records early, test inputs, quarantine and reprocess safely.

  3. Dependency Failures Throttled services, full storage, or region-level outages. Fix: monitor external dependencies, plan for burst capacity, load balance.

  4. Unexpected Load Spikes Sudden surge causes pipelines to overrun quota or resources. Fix: autoscaling with caps, smart throttling, plan emergency resource pools.

  5. Single Region Risk Region outage stops everything. Fix: multihomed design with auto-failover and central work distribution.

🎧 Spotify's Event Delivery System

Spotify records 100B+ user events daily: song plays, playlist follows, ads clicked.

They need:

  • Timely data

  • Accurate event delivery

  • Fair artist payments based on events

So they built a system using:

  • Google Cloud Pub/Sub for ingest

  • ETL microservices (extract, partition, deduplicate)

  • Google Cloud Storage for hourly buckets

⚖️ Spotify's 3 Key SLOs

  1. Timeliness: How quickly does an hourly bucket get delivered after the hour ends?

  2. Completeness: What percentage of events published actually get delivered?

  3. Skewness: Are events being delivered in the right hourly bucket?

🚧 Incident Handling & Operations

Spotify separates:

  • Monitoring (CPU, memory, alerts)

  • SLOs (timeliness, completeness)

  • Logging (minimal, relevant)

CI/CD process:

  1. Unit + integration tests

  2. Staging deployment

  3. Canary to small production segment

  4. Full deploy after approval

During incidents: rollback first, diagnose later.

🔄 Recovery & Reprocessing

When things go wrong:

  • Incomplete data → Redeliver from last known good checkpoint

  • Skewed data → Reshuffle to correct bucket

  • Corrupt outputs → Notify teams, reprocess safely

Spotify encourages clear ownership, minimal exposure, and fast replays.

🔙 TL;DR

  • Use checkpointing to save state and recover fast

  • Build reusable components to scale pipelines safely

  • Score your pipeline with the maturity matrix

  • Monitor for data delay, skew, and loss

  • Learn from Spotify: simple APIs, clear SLOs, and CI/CD guardrails
