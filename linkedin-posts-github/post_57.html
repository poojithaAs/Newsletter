#EPISODE2 THE LOG THAT ATE THE CLOUD We #BrokeIt Together

Sara Baqla (#Devloper): "I added some debug logs to track a payment issue."

Poojitha A S (#Devops/ #SRE): "What are you logging?"

Sara Baqla: "User ID, session ID, request metadata. Normal debug info."

Poojitha A S: "On every request? We&#x27;re around 45–50K RPS."

Sara Baqla: "The platform samples logs automatically… right?"

Poojitha A S: "It does. I approved the PR assuming that was enough." #BOTH: "Okay… let&#x27;s fix it together ." WHAT ACTUALLY HAPPENED 10:18 : Deploy goes out. Canary at 15% traffic looks healthy. 10:24 : Datadog shows log volume climbing. Alert fires (Slack). 10:27 : Alert acknowledged. No page. No rollback. 10:32 : Traffic ramps.

Logs scale linearly with traffic. 10:36 : Each pod is now writing hundreds of MBs per minute. 10:40 : Autoscaling adds pods, which adds… more logs. 10:43 : Nodes hit disk pressure. ENOSPC errors start. 10:47 : Checkout errors appear across regions. 11:02 : Rollback begins. 11:17 :System stabilizes.

THE NUMBER WE DIDN&#x27;T DO

• ~45,000 requests/sec

• ~2–3 KB per log line

• ≈ 90–130 MB/sec

• ≈ 5–8 GB/min per region Multiply by pods. Multiply by regions. Multiply by autoscaling. At small scale, this is a log. At 50K RPS, it&#x27;s infrastructure. #DAMAGE

• Short outage window

• Checkout failures

• Seven-figure revenue impact

• Unexpected logging bill

• One very uncomfortable postmortem THE

#FIX ✅ Log sampling enforced at 1% in prod ✅ PR checklist: "log volume at 100% traffic?" ✅ Disk alerts fire earlier ✅ Cost-impact alerts now escalate ✅ Fewer alerts, higher signal THE

#LESSON This wasn&#x27;t a logging bug. It was a scale + toil problem. Reliability isn&#x27;t about stopping mistakes. It&#x27;s about making normal decisions safe at scale. WHOSE #FAULT? Sara Baqla wrote the logs. Poojitha A S approved the PR. The system allowed it. So we fixed the system.
